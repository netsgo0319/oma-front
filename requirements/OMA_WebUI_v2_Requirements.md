# OMA WebUI v2 요구사항 명세서

> Oracle Modernization Accelerator — 차세대 웹 인터페이스 기능 요구사항  
> 마이그레이션 전 과정을 에이전트 기반으로 제어·추적·확인하는 통합 플랫폼

---

## 1. 전체 구조 개요

v2는 마이그레이션의 3대 영역을 **하나의 흐름**으로 통합 관리한다.

| 영역 | 설명 | 핵심 방식 |
|------|------|----------|
| **데이터베이스 마이그레이션** | 스키마·프로시저·펑션·뷰 등 DB 객체 변환 | DMS SC 규칙 기반(~92%) + AI 에이전트 잔여 처리(~8%) |
| **애플리케이션 마이그레이션** | MyBatis/iBatis SQL 추출 → 변환 → 테스트 → 병합 | AI 에이전트 기반 쿼리 추출·변환·자동 재시도 |
| **데이터 마이그레이션** | 실제 데이터를 Oracle → PostgreSQL로 이관·검증 | DMS Full Load + 건수·데이터 검증 |

### 메뉴 구조 (6개 대메뉴)

```
📊 대시보드
🗄️ 데이터베이스 마이그레이션
📱 애플리케이션 마이그레이션
📦 데이터 마이그레이션
🔧 도구
⚙️ 설정
```

## 2. 📊 대시보드

마이그레이션 전체 현황을 한눈에 파악하는 메인 화면이다.

### 2.1 마이그레이션 진행 현황 보드

- 3대 영역(DB / 앱 / 데이터)의 진행률을 프로그레스 바로 표시
- 각 영역의 단계별 완료/진행중/대기 상태를 시각적으로 표시 (예: ✅ → 🔄 → ⏳)
- 전체 소요 시간 및 예상 남은 시간 표시

### 2.2 워크플로우 다이어그램

- 전체 마이그레이션 프로세스를 인터랙티브 플로우차트로 표시
- 현재 진행 중인 단계를 하이라이트
- 각 노드 클릭 시 해당 메뉴로 이동

### 2.3 실시간 작업 상태

- 현재 실행 중인 에이전트/작업 정보 (이름, PID, 경과 시간)
- 최근 완료된 작업 히스토리 (최근 10건)
- 에러/경고 알림 요약

### 2.4 핵심 지표 카드

| 지표 | 내용 |
|------|------|
| DB 객체 변환율 | DMS SC 자동 변환 N개 / 에이전트 변환 N개 / 실패 N개 |
| SQL 변환율 | 변환 완료 N개 / 재작성 필요 N개 / 미처리 N개 |
| 테스트 통과율 | PASS N개 / FAIL N개 / SKIP N개 |
| 데이터 이관율 | 이관 완료 테이블 N개 / 총 행 수 / 검증 결과 |

## 3. 🗄️ 데이터베이스 마이그레이션

Oracle DB 객체를 PostgreSQL로 변환하는 과정이다. DMS Schema Conversion을 우선 실행하고, 변환 실패한 객체만 AI 에이전트가 처리한다.

### 3.1 DMS Schema Conversion 실행

DMS SC의 6단계 파이프라인을 제어하고 결과를 확인하는 화면이다.

**기능:**

- DMS SC 6단계를 순차 또는 일괄 실행
  1. Extension Pack 설치
  2. Oracle 메타데이터 임포트
  3. 스키마 변환 (Oracle → PostgreSQL DDL)
  4. 변환 DDL 스크립트 S3 내보내기
  5. 변환 DDL을 PostgreSQL에 직접 적용 (export_to_target)
  6. Assessment 보고서 생성
- 각 단계별 진행 상태 표시 (대기/실행중/완료/실패)
- 실행 로그 실시간 표시
- 단계별 개별 재실행 기능

### 3.2 DMS SC 결과 확인

DMS SC가 생성한 Assessment 보고서와 변환 결과를 확인하는 화면이다.

**기능:**

- Assessment CSV 파일 파싱 및 테이블 표시
- 객체별 변환 복잡도 분류: Simple / Medium / Complex
- 트리아지 결과 요약
  - **DMS 자동 변환 완료**: 테이블, 인덱스, 시퀀스, 제약조건, 기본값 등
  - **에이전트 변환 필요**: 저장 프로시저, 복잡한 뷰(CONNECT BY 등), 패키지, 펑션
- 객체 유형별 통계 차트 (변환 성공/실패 비율)
- 변환 실패 객체 목록 및 실패 사유 표시

### 3.3 AI 에이전트 스키마 변환

DMS SC가 처리하지 못한 잔여 객체를 AI 에이전트로 변환한다.

**기능:**

- 에이전트 변환 필요 객체 목록 표시 (3.2의 트리아지 결과 기반)
- 에이전트 파이프라인 실행: Discovery → Schema Architect → Code Migrator → QA Verifier → Evaluator
- 각 에이전트 단계별 진행 상태 및 로그 표시
- 변환 결과 객체별 확인
  - Oracle 원본 DDL(PL/SQL)과 PostgreSQL 변환 DDL(PL/pgSQL) 좌우 비교
  - 적용된 변환 규칙 목록 (예: CONNECT BY → WITH RECURSIVE, MERGE → INSERT ON CONFLICT)
- Evaluator GO/NO-GO 판정 결과 (100점 만점 스코어)
- Remediation 루프 이력 (최대 3회 재시도 내역)

### 3.4 PostgreSQL 스키마 검증

변환된 스키마가 PostgreSQL에 정상 적용되었는지 검증한다.

**기능:**

- Oracle 소스와 PostgreSQL 타겟 간 객체 1:1 대조
  - 테이블, 컬럼, 인덱스, 제약조건, 시퀀스, 프로시저, 펑션, 뷰 각각 비교
- 누락/불일치 객체 하이라이트
- PostgreSQL 메타데이터 조회 (information_schema 기반)
- 전환된 스키마에 대한 이기종 DB 대응 방법 참고 정보 표시
  - 패키지 → 스키마+함수, 프로시저 → PL/pgSQL, 펑션 변환 패턴, 디비링크 대응 등

### 3.5 변환 컨텍스트 관리

DB 변환 과정에서 생성된 정보를 앱 마이그레이션 단계로 전달하기 위해 관리한다.

**기능:**

- 변환 컨텍스트 자동 수집 및 저장
  - 전환된 스키마 목록 및 매핑 정보
  - 패키지 → 스키마+함수 변환 매핑 테이블
  - 프로시저/펑션 이름 변경 이력
  - 디비링크 제거/대체 내역
  - 데이터타입 변환 매핑 (NUMBER → NUMERIC, VARCHAR2 → VARCHAR 등)
- 컨텍스트 데이터를 조회·편집할 수 있는 뷰
- 앱 마이그레이션 단계에서 이 컨텍스트를 자동 참조

## 4. 📱 애플리케이션 마이그레이션

MyBatis/iBatis 기반 Java 애플리케이션의 SQL을 추출하고, PostgreSQL용으로 변환하며, 테스트를 거쳐 최종 병합하는 과정이다.

### 4.1 Mapper 파일 탐색 및 분석

프로젝트 소스코드에서 MyBatis/iBatis Mapper XML 파일을 찾아 분석한다.

**기능:**

- 지정된 소스 경로에서 Mapper XML 파일 자동 탐색
  - 실무에서 mapper가 아닌 이름으로 사용되는 경우도 탐색 대상에 포함
- 탐색 결과를 트리 뷰 또는 테이블로 표시
  - 파일 경로, SQL 구문 수, SQL 유형(select/insert/update/delete) 통계
- 각 Mapper 파일 클릭 시 XML 내용 미리보기
- JNDI 데이터소스 설정 분석 결과 표시

### 4.2 SQL 추출

Mapper XML 파일에서 SQL 구문을 추출한다. MyBatis의 복잡한 구조를 처리해야 한다.

**기능:**

- Mapper XML 파싱 및 SQL 추출
  - 처리해야 하는 복잡 요소: `resultMap`, `collection`, `sql/include` (재사용 SQL 프래그먼트), 동적쿼리 태그 (`foreach`, `if`, `where`, `choose`, `when`, `otherwise`, `set`, `trim`, `bind`), `association`, `CDATA` 섹션
  - OGNL 표현식이 포함된 동적 SQL은 단순 파싱이 어려우므로 별도 처리
- 추출 방식 2가지 제공 (선택 또는 병행)
  - **정적 추출**: XML 파서 기반으로 SQL 태그 직접 추출 (빠르지만 동적 SQL 한계)
  - **동적 추출 (Mock 테스트 에이전트)**: MyBatis SqlSessionFactory를 활용하여 실제 쿼리를 수행시켜 완성된 샘플 쿼리 수집
    - 실제 DB 연결 없이 목 테스트로 쿼리를 뽑아보는 에이전트
    - 쿼리뿐 아니라 액션(insert/update/delete)도 trajectory로 기록하여 마이그레이션 이후 비교 가능
- 추출된 SQL 목록을 테이블로 표시
  - SQL ID, SQL 유형, 원본 SQL, 동적 태그 포함 여부, 복잡도 등급
- 추출 결과를 CSV로 다운로드

### 4.3 SQL 변환 대상 필터링

추출된 SQL 중 실제 재작성이 필요한 것만 필터링한다.

**기능:**

- DB 변환 컨텍스트(3.5)를 참조하여 자동 필터링
  - Oracle 고유 문법 사용 SQL 자동 식별 (NVL, DECODE, SYSDATE, ROWNUM, CONNECT BY, MERGE 등)
  - 스키마 접두사 사용 SQL 식별
  - 패키지.프로시저 호출 SQL 식별
  - 디비링크 참조 SQL 식별
- 필터링 결과를 3가지로 분류
  - **변환 필요**: Oracle 고유 문법이 포함된 SQL
  - **신규 쿼리**: 기존에 없던 새로운 쿼리 (변경된 스키마 구조에 맞춰 새로 작성 필요)
  - **변환 불필요**: 표준 SQL로 양쪽 DB에서 동일하게 동작
- 필터링 결과를 편집 가능한 테이블로 표시 (수동 분류 변경 가능)
- 필터링 기준을 사용자가 커스텀 추가 가능

### 4.4 쿼리 재작성 (AI 에이전트)

변환 필요 SQL을 AI 에이전트가 자동으로 PostgreSQL용으로 재작성한다.

**기능:**

- 재작성 에이전트 파이프라인 실행
  1. **변환**: Oracle SQL → PostgreSQL SQL 변환 (4-Phase 방법론 적용)
  2. **테스트케이스 생성**: 변환된 쿼리에 대한 테스트케이스 자동 생성
     - 바인드 변수 추출 및 샘플 데이터 매핑
     - 다양한 변수 조합 (특히 NULL, 빈 문자열, 경계값)
  3. **실행 및 비교**: Oracle과 PostgreSQL에서 동일 입력으로 실행 → 결과 비교
  4. **자동 재시도**: 결과가 다른 경우 AI가 쿼리를 수정하여 재시도 (최대 N회)
  5. **MyBatis 문법 적용**: 변환 완료된 SQL을 MyBatis XML 문법에 맞게 재작성
- 실행 모드 선택
  - **샘플 변환**: 소수 SQL만 선별하여 먼저 변환 (결과 사전 검증 용도)
  - **전체 변환**: 변환 필요 SQL 전체 일괄 변환
- 변환 진행률 실시간 표시 (전체 N개 중 M개 완료, 성공/실패/재시도 현황)
- 개별 SQL 변환 결과 상세 확인
  - Oracle 원본 ↔ PostgreSQL 변환 SQL 좌우 비교 (diff 하이라이팅)
  - 적용된 변환 규칙 목록
  - 테스트 결과 (PASS/FAIL/재시도 이력)
  - 에이전트의 변환 판단 근거 (trajectory 로그)

### 4.5 변환 결과 수동 검수 및 수정

에이전트가 처리하지 못했거나 사용자 확인이 필요한 SQL을 수동으로 검수한다.

**기능:**

- 변환 실패/재시도 초과 SQL 목록 표시 (에이전트가 N회 재시도 후에도 실패한 건)
- SQL 편집기 제공
  - 변환된 SQL을 직접 수정 가능
  - 수정 후 즉시 PostgreSQL 구문 검증 (syntax check)
  - 수정 후 즉시 Oracle/PostgreSQL 양쪽 실행 비교 가능
- 커스텀 쿼리 실행 기능
  - 임의의 SQL을 Oracle 또는 PostgreSQL에 직접 실행
  - 실행 결과 및 실행 계획(EXPLAIN) 확인
- 웹 셸 터미널 (v1 기능 유지)

### 4.6 XML 병합

변환 완료된 SQL을 원본 MyBatis XML Mapper 파일에 병합하여 최종 파일을 생성한다.

**기능:**

- 병합 실행 (기존 타겟 XML 삭제 → 변환 SQL 병합)
- 병합 시 주의 사항 처리
  - 쿼리가 분리되어야 하거나 VO를 재작성할 필요가 있을 경우 → Java 애플리케이션 코드(VO, DAO 등) 변경 필요 부분을 별도로 리포팅
  - MyBatis XML 구조(CDATA, 동적 태그, 네임스페이스 등) 보존 확인
- 병합 결과 확인
  - 원본 XML과 병합 XML의 diff 비교
  - XML 유효성 검사 (well-formed, MyBatis DTD 기준)
- 병합된 파일 다운로드

### 4.7 애플리케이션 테스트 지원

변환된 애플리케이션을 실제 동작시켜 검증하는 과정을 지원한다.

**기능:**

- 사용자 테스트 가이드 제공
  - Spring AOP 기반 에러 로깅 설정 방법 안내
  - Aspect를 Repository 레벨 예외 발생 시 특정 스토리지(파일시스템, CloudWatch, S3 등)에 에러로그를 기록하도록 설정
- 테스트 결과 에러로그 수집 및 표시
  - 에러로그 파일을 업로드하거나 경로를 지정하여 자동 수집
  - 에러 유형별 분류 및 통계 (SQL 오류, 타입 불일치, 널 처리 등)
  - 에러가 발생한 SQL ID → 원본 Mapper 파일 역추적
- 자동 수정 에이전트
  - 수집된 에러로그를 기반으로 AI 에이전트가 해당 SQL을 자동 수정
  - 수정 내역 확인 및 승인/반려 기능
  - 수정 후 XML 재병합 자동 실행

## 5. 📦 데이터 마이그레이션

Oracle에서 PostgreSQL로 실제 데이터를 이관하고 검증한다.

### 5.1 데이터 이관 실행

- DMS Full Load 또는 직접 전송 방식으로 데이터 이관
- 테이블별 이관 상태 표시 (대기/진행중/완료/실패)
- 이관 진행률 표시 (전체 테이블 수, 전체 행 수 기준)
- 실행 로그 실시간 표시

### 5.2 데이터 검증

- Oracle ↔ PostgreSQL 테이블별 행 수 비교
- 샘플 데이터 비교 (랜덤 샘플링)
- 시퀀스 현재 값 비교
- 검증 결과를 PASS/FAIL 테이블로 표시
- 불일치 항목에 대한 상세 정보 드릴다운

## 6. 🔧 도구

마이그레이션 과정을 보조하는 유틸리티 기능 모음이다.

### 6.1 SQL 실행기

- Oracle 또는 PostgreSQL을 선택하여 임의의 SQL 실행
- 실행 결과를 테이블로 표시
- EXPLAIN ANALYZE 실행 (실행 계획 확인)
- 양쪽 DB에 동일 SQL을 동시 실행하여 결과 비교

### 6.2 실시간 로그 뷰어

- 현재 실행 중인 작업의 로그를 실시간 표시
- ANSI 컬러 코드 제거 및 포맷팅
- 로그 검색 및 필터링

### 6.3 에이전트 로그 뷰어 (qlog)

- AI 에이전트 변환 작업 전용 로그 뷰어
- 에이전트별 대화 내역 (trajectory) 확인
- 에이전트가 사용한 도구 호출 이력 표시
- 변환 판단 근거 추적

### 6.4 변환 보고서

- AI 에이전트를 활용한 종합 보고서 자동 생성
  - Executive Summary, 스키마 인벤토리, DMS SC 결과, AI 변환 내역
  - DDL 실행 결과, 데이터 이관 결과, 검증 결과
  - Evaluator 점수 (5차원), GO/NO-GO 판정
  - 리스크 평가, Remediation 이력, 권고사항
- 생성된 보고서(HTML) 웹 화면 내 조회
- 보고서 다운로드

### 6.5 변환 패턴 지식베이스

- 에이전트가 학습한 변환 패턴 조회 (DynamoDB 패턴 메모리)
- Oracle 구문 → PostgreSQL 변환 규칙 레퍼런스 검색
- 사용자가 커스텀 변환 규칙 추가/수정

## 7. ⚙️ 설정

### 7.1 프로젝트 설정

- 프로젝트 생성 및 선택
- 프로젝트별 환경 변수 관리
  - OMA 기본 경로, 애플리케이션 소스 경로
  - 소스 DB(Oracle) 접속 정보
  - 타겟 DB(PostgreSQL) 접속 정보
  - DMS 설정 (Migration Project ARN, S3 버킷)
- 환경 변수 테이블 조회 (비밀번호 마스킹)
- DB 접속 테스트 기능

### 7.2 에이전트 설정

- AI 모델 설정 (Bedrock 모델 ID, 리전)
- 에이전트별 파라미터 조정
  - Remediation 최대 재시도 횟수 (기본 3회)
  - Evaluator GO/NO-GO 임계점수 (기본 70점)
  - 쿼리 재작성 최대 재시도 횟수
- 토큰 예산 모니터 설정 (Context Budget)

### 7.3 테스트 설정

- SQL 단위 테스트 바인드 변수 파라미터 설정
- 테스트 데이터 소스 설정 (Oracle 샘플 데이터 / 합성 데이터 / 사용자 제공)
- Spring AOP 에러 로깅 수집 경로 설정

## 8. 공통 기능 요구사항

### 8.1 진행상황 추적 (Tracking)

- 모든 마이그레이션 단계의 진행상황을 실시간으로 추적
- 대시보드에서 전체 현황을 한눈에 확인
- 각 단계별 상세 페이지에서 세부 진행 현황 확인
- 중간중간 생성되는 결과물(CSV, HTML, XML, JSON)을 웹에서 직접 조회 및 다운로드

### 8.2 작업 관리

- 동시에 하나의 장시간 작업만 실행 가능 (중복 실행 방지)
- 작업 실행 중 다른 메뉴 탐색은 자유롭게 허용 (UI 블로킹 없음)
- 모든 작업에 대해 중단 기능 제공
- 작업 완료/실패 시 알림
- 작업 이력 관리 (과거 실행 내역 조회)

### 8.3 가시성

- 모든 기능은 웹 화면에서 시각적으로 확인 가능해야 함
- 에이전트 내부 동작(도구 호출, 판단 과정)도 투명하게 노출
- 에러 발생 시 원인과 해당 SQL/객체를 즉시 식별 가능
- 변환 전후 비교(diff)를 모든 관련 화면에서 제공

### 8.4 커스텀 쿼리 지원

- 사용자가 임의의 SQL을 직접 작성하여 Oracle 또는 PostgreSQL에 실행 가능
- 특정 단계에 종속되지 않고, 도구 메뉴에서 항상 접근 가능
- 실행 결과를 테이블/JSON으로 확인
- Oracle과 PostgreSQL 결과를 나란히 비교하는 모드 제공
